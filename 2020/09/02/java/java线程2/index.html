java并发包、线程池原理分析、锁的深度化

  • Vector和ArrayList区别

实现原理都是通过数组实现-查询速度快、增加、修改、删除速度慢
区别：线程安全问题
Vector是安全、ArrayList线程不安全。 ArrayList效率高
vector的add方法使用synchronized锁(同步函数)，多个线程操作时，由于争取资源，所以效率低一些

  • HasTable和HasMap

hasTable线程安全，hasMap线程不安全
hasMap底层：链表+数组(链表增加、修改、删除快，数组查询快) put方法：通过hascode取模得到下标位置(一致性取模算法)
hasTable的put()方法使用synchronized(同步函数)
collections中.synchronizedMap(Map<k,v>)—将不安全的map转为安全的map

ThreadLock

  • 什么是ThreadLock？

为每一个线程提供一个局部变量，使线程互不影响 即便共享同一个全局数据

  • ThreadLock原理是什么？

    通过map存储所以线程以及其自己的变量

ConcurrenHasMap

分段锁计算

多线程共享同一个HasTable，效率非常低(因为锁的资源竞争)。加锁后，每次只能有一个线程去操作hastable。
ConcurrenHasMap通过分段锁(最高为16段)来提高操作的速度。即将一个整体拆分成多个小的HasTable，即有多个锁，多线程操作时，由于操作的是不同的下标即操作不同的锁，所以可以提高多线程操作效率(即减少了锁的资源竞争)

CountDownLatch(计数)

作用：计数用的，通过await()方法让某些线程在后面执行。jdk1.5的并发包出现

  • new CoountDownLatch(线程总数) 创建一个countDownlatch对象
  • **.countDown()**线程执行一次，线程总数减1
  • **.await()**countDownLatch中不为0，则一直等待

CyclicBarrier(计数)

作用：计数用的，当线程达到一定次数，所有线程才并行执行

  • **.await()**所有线程没有执行完前面的代码，则后面代码不会执行，线程一直在此等待 但线程要使用同一个CyclicBarrier对象

Semaphore(信号量)

作用：信号量为0后,其余线程都要等某个获得资源权限的线程释放资源，然后等待的线程谁抢到释放的资源谁使用

  • .acquire() 获取到资源权限，总数减1
  • .release() 释放资源

一个信号量，当信号量为0后，其余线程都要等某个获得资源权限的线程释放资源，然后谁抢到释放的资源谁使用
例子：3个茅坑10个人用。

并发队列

  • 生产者消费者概念
  • 队列(消息中间件)：多个线程共享数据
  • 并发队列有界、无界（区别：存储长度一个有限制，一个无限制）
  • 阻塞队列、非阻塞队列 (阻塞：生产者写入满时，生产者就会进入到阻塞–当队列为空的时候，消费者会等待)

队列遵循的规则–先进先出，后进后出

  • 无界限 如：ConcurrentLinkedDeque
  • 可阻塞，有界限 如：BlockingQueue(子类:1-ArrayBlockingQueue 2-LinkedBlockingQueue)

* 线程池

  • 什么是线程池？为什么使用线程池？

经常创建、启动、销毁非常耗资源，使用线程池进行线程管理、复用。可以提高程序效率
线程池好处：1-重复利用，降低资源消耗 2-提高响应速度 3-提高线程可管理性–线程池进行创建和分配

  • 线程池使用方式？

接口【Executor】封装好了四种线程池
【1-newCachedThreadPool】创建一个可缓存线程池，如果线程长度超过处理需要，可灵活回收空闲线程，若无可回收，则新建线程。 优点：可重复利用
【2-newFixedThreadPool】创建一个定长线程池，可控制线程最大并发数，超出的线程会在队列中等待。
【3-newScheduledThreadPool】创建一个定长线程池，支持定时及周期性任务执行。
【4-newSingleThreadExecutor】创建单线程化的线程池，它只会用唯一的工作线程来执行任务，保证所有任务按照指定顺序(FIFO、LIFO，优先级)执行。

  • 线程池主要是配置返回了ThreadPoolExecutor，其构造函数参数如下：

CodeOllSize–核心线程池大小
MaiMumPoolSize–线程池大小
keepAliveTime–终止时间 Unit–超时秒数
线程池包括：核心线程池和最大线程池

线程数合理配置

  • 遵循两个原则

1-cpu密集(cpu频繁调度)：cpu密集时，任务少配置线程数，线程数和大概cpu核数相同，这样可以使每个线程都在执行任务
2-io密集：io密集时，大部分线程都阻塞，因此需要多配置线程数，2*cup核数

java锁机制

  • 悲观锁

    默认包含排它锁，每次拿数据的时候，都会上锁 缺点：效率低，因为只能保证一个连接进行操作。

  • 乐观锁

    版本标识–即cas无锁机制

  • 重入锁

    重入锁(锁可以传递给下一个方法)与非重入锁(会产生死锁)

  • 读写锁

    可以共读，不可共写。读写锁最好一起使用

  • CAS无锁

    cas无锁机制效率比有锁机制高，cas无锁机制其实和乐观锁类似概念。
    cas三个参数：1-V表示要更新的变量 2-E表示预期值 3-N表示新值
    仅当v值=e值是，才会将v值设为n，如v值和e值不同，说明已经有其他线程做了更新，则当前线程说明都不做。最后，cas返回当前v的真实值

  • 自旋锁

    底层通过cas无锁机制无限循环实现。

  • 排它锁
  • 分布式锁
